The Problem

⚠
Tool schemas bloat the context
Every MCP tool carries a full JSON Schema definition. With 30+ tools from Calendar, GitHub, and Search servers, that's ~15,000 tokens of schemas injected into every single turn. Those tokens are billed every time.
⚠
Results accumulate in context
Each tool call adds its result to the conversation. A 7-step task means 7 results growing the context quadratically. By tool call #7, the LLM is re-processing all previous results.
⚠
No parallel execution
Traditional tool calling is sequential — one call per turn. Independent operations (search + calendar check) can't run simultaneously. Each round-trip adds latency.

Measured impact

Same task, same LLM (Claude Opus), same MCP servers. Direct tool calling: $6.95 (41K first-turn context, 15K tokens of schemas). With codemode: $3.25 (26K first-turn context, 221 tokens of schemas). 53% cost reduction.

The Solution

An MCP server that wraps all your tools behind a single run_python tool. The agent writes Python code, and the code runs inside an isolated sandbox with access to all tools via RPC.

How It Works

1
MCP client starts the server
Any MCP-compatible client (Claude Code, Cursor, custom agents) reads .mcp.json and spawns server.py as a child process. Communication is stdio JSON-RPC.
2
Registers MCP server configs (no connections yet)
Reads *.json config files from $CODEMODE_MCPS_DIR. Registers each server's connection details but does NOT connect. Startup is instant — zero network calls.

3
Exposes a single tool: run_python
Agent sees one tool with server names listed. No tool schemas, no tool names — the agent discovers tools at runtime using _discover() and _schema() inside the sandbox.
4
Agent calls run_python with servers=[...]
The agent passes servers=["Calendar", "GitHub"] to load only the servers it needs. On first use, the server connects (SSE handshake or stdio subprocess), loads tools, and injects them into the sandbox. Subsequent calls with the same server are instant (cached).
# Agent passes servers=["Calendar", "GitHub"]
# Calendar connects NOW, GitHub connects NOW
# Sandbox container also starts on first call

# Discover what's available
tools_list = await _discover()
schema = await _schema('list-events')

# Call tools
cals = await tools['list-calendars']()
repos = await tools['search_repositories'](query='test')
print(json.dumps({"cals": len(cals), "repos": repos['total_count']}))
5
Tool calls cross the boundary via RPC
When code calls tools['create-event'](...), it triggers a JSON-RPC request from the container to the host. The host routes through ToolProxy to the real MCP server, unwraps the response, and sends it back.
6
Result returned, state persists
Captured print() output returned to the agent. Variables persist for the next call. Servers stay connected — no reconnect overhead. Servers never used in this session are never connected.

Sandbox Architecture

What's baked into the container and how isolation works

⚙
StreamProxy
Replaces sys.stdout/stderr. Every print() becomes a JSON message sent to the host for capture.
🔗
RPC Infrastructure
Future-based request/response over stdin/stdout. Each tool call gets a unique ID, 30s timeout prevents deadlocks.
📚
_ToolsDict
Dict with __missing__ auto-proxy. Known tools pre-loaded; on-demand tools get RPC proxies automatically on first access.
🔎
Baked Metadata
Full tool schemas baked as JSON into the entrypoint. _discover() and _schema() read locally — zero RPC for known tools.
🏭
Server Proxies
mcp_calendar.list_events() — _MCPProxy groups tools by server via __getattr__ routing.
📊
Persistent Namespace
A dict with tools, safe modules (json, datetime, re, math, asyncio), discovery helpers, and all user variables from previous calls.

Response unwrapping

ToolProxy automatically normalizes MCP responses so the sandbox code gets clean Python objects.

1 Normalize calling conventions — dict arg, kwargs, positional args all accepted
2 Strip None values — LLMs often pass optional params as None
3 Unwrap MCP content blocks — [{type:"text", text:"{}"}] → {}
4 Parse JSON strings — '{"id":"123"}' → {"id": "123"}

Example — Multi-Tool Task in One Code Block

Task: Find AI conferences, check calendar for conflicts, schedule an event, create a GitHub repo

import json

# Search for conferences and check calendar in parallel
import asyncio
conferences, freebusy = await asyncio.gather(
    tools['google_search'](q='top AI conferences March 2026'),
    tools['get-freebusy'](
        calendars=[{"id": "primary"}],
        timeMin='2026-03-16T00:00:00',
        timeMax='2026-03-19T23:59:59'
    )
)

# Pick the top result and find a free slot
top = conferences['organic'][0]
print(f"Selected: {top['title']}")

# Schedule the event
event = await tools['create-event'](
    calendarId='primary',
    summary=top['title'],
    start='2026-03-17T10:00:00',
    end='2026-03-17T11:00:00'
)

# Create a repo and add a README
repo = await tools['create_repository'](name='conference-notes')
await tools['create_or_update_file'](
    owner=repo['owner']['login'], repo='conference-notes',
    path='README.md', content=f'# {top["title"]}\nNotes from the conference',
    message='Initial commit'
)

print(json.dumps({
    "event": event['summary'],
    "repo": repo['html_url']
}, indent=2))

What just happened

• 6 tool calls across 3 MCP servers (Search, Calendar, GitHub) in one code block
• asyncio.gather() ran search + calendar check in parallel
• Results from one tool used as input to the next — no round-trips to the LLM
• Agent context window: ~221 tokens of tool schemas instead of ~15,000
• All data stayed inside the sandbox — only print() output returned to the agent
